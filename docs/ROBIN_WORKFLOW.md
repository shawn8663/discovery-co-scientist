# Robin-Inspired Therapeutic Discovery Workflow

Discovery Co-Scientist includes a `therapeutic_discovery` workflow profile that
imports the strongest architectural ideas from Robin into the existing durable
co-scientist app.

## Flow

1. A researcher provides a target disease or therapeutic goal.
2. The Assay agent proposes biologically relevant, fast, feasible cell-culture
   assay strategies.
3. The Assay agent evaluates each assay and ranks the bounded set with BTL-style
   scores when pairwise comparisons are available.
4. The Candidate agent synthesizes a candidate-generation goal from the winning
   assay, proposes single-agent therapeutic candidates, evaluates them, and
   ranks the bounded candidate set.
5. The Analysis agent registers uploaded experimental datasets and records a
   controlled analysis run artifact.
6. The Result Interpreter converts analysis summaries into structured
   experiment insights.
7. The Candidate agent can regenerate a new candidate round from those insights.

## Agent Roles

- `assay`: Generates assay proposals, detailed assay evaluations, and fixed-set
  assay rankings.
- `candidate`: Generates therapeutic candidates, evaluates them, ranks them,
  and regenerates candidates from experiment insights.
- `analysis`: Creates controlled analysis-run artifacts for flow cytometry,
  RNA-seq, and tabular data.
- `result_interpreter`: Converts analysis results into positive hits, negative
  hits, suggested mechanisms, follow-up assay outlines, and constraints.

## Prompt Stack

Robin prompt templates live in `config/prompts/robin_*.md`. They cover query
generation, assay generation/evaluation/judging, candidate
generation/evaluation/judging, flow cytometry analysis, RNA-seq analysis, and
result interpretation.

The candidate and judge prompts explicitly preserve Robin's core criteria:
target validation, disease/model relevance, mechanistic specificity,
experimental feasibility, safety/tolerability, ADME/PK and delivery feasibility,
and novelty balanced against evidence.

## Data and Artifacts

Robin workflow rows are stored in SQLite tables for assays, candidates,
evaluations, analysis runs, and experiment insights. Each row points to a JSON
artifact under `data/artifacts/<session_id>/...`, and analysis outputs are also
registered in the session workspace manifest.

Therapeutic discovery sessions use the shared evidence-bundle workflow before
assay and candidate generation. Uploaded project files are searched first,
Paperclip is the first external source when configured, and the workflow is
treated as clinical/translational so ClinicalTrials.gov is included. The
resulting `evidence_bundle` artifact gives Robin agents a traceable
`canonical_evidence` index plus links back to raw `retrieved_literature`
artifacts.

## Human Decisions

Assays and candidates can be pinned or rejected by expert feedback. The web UI
posts those decisions through the shared feedback endpoint with a Robin target
type, records the human decision in `system_feedback`, and updates the assay or
candidate state for later scheduling and review.

## Benchmarks

The offline fixture registry includes `robin-dry-amd`, a dry AMD therapeutic
discovery fixture with mock retrieval and model outputs for assay generation,
assay evaluation, candidate generation, and candidate evaluation. The companion
`aml-general-compatibility` fixture keeps the original general-hypothesis
workflow covered while the Robin workflow evolves. Benchmark reports include
quality, cost, latency, duplicate rate, retrieval hits, and ranking agreement.

## Safety

The workflow treats generated assays and candidates as research hypotheses and
experimental outlines. Precise executable wet-lab protocols are not generated by
default. Analysis execution is represented as a controlled science-skill wrapper
with provenance rather than unrestricted arbitrary code execution.
